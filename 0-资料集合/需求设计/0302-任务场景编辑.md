# 反辐射无人机仿真系统——协同任务规划模块任务场景编辑功能需求描述

## 1. 功能需求描述

| **项目**         | **描述** |
|------------------|----------|
| **需求名称**     | 任务场景编辑 |
| **需求标识**     | FR-COOP-01 |
| **需求概述**     | 具备图形化的任务场景编辑功能，结合地理信息系统，编辑协同作战任务背景、任务目标、威胁要素、兵力部署、作战行动等内容，支持用户通过可视化方式快速构建反辐射无人机协同作战任务场景。 |
| **需求来源**     | 系统总体设计要求 / 协同任务规划模块能力指标 |
| **前置条件**     | 1. 地理信息系统基础数据已加载（包含地形、影像、矢量图层等）。<br>2. 模型库中存在可用的实体模型（无人机、目标、威胁等）。<br>3. 用户已通过身份认证并具备场景编辑权限。<br>4. 系统图形化编辑引擎已初始化。 |
| **触发条件**     | 1. 用户进入“任务场景编辑”界面，新建或打开已有场景。<br>2. 通过工具栏点击“新建场景”或“打开场景”按钮。<br>3. 从任务管理界面跳转至场景编辑。 |
| **退出条件**     | 1. 用户保存场景并退出编辑界面。<br>2. 用户放弃修改直接退出。<br>3. 发生不可恢复错误，模块退出并记录错误。 |
| **功能参与者**   | 用户（任务规划人员）、地理信息系统服务、模型库、场景数据库 |
| **输入**         | 地理信息图层选择、实体放置（拖拽）、实体属性设置、路径绘制、作战行动定义、场景元数据（名称、描述、时间等）。 |
| **输出**         | 完整的任务场景文件（包含地理信息、实体部署、行动规划等），可用于仿真推演、兵力需求计算、任务分配等后续模块。 |
| **处理流程**     | 1. **场景初始化**：用户选择新建场景或打开已有场景。若新建，设置场景基本信息（名称、区域、时间等）；若打开，从数据库加载场景数据。<br>2. **地理信息加载**：根据场景区域加载对应地理信息（地形、影像、矢量图层），支持图层切换和叠加。<br>3. **实体部署**：<br>   a. 用户从模型库拖拽实体模型至地图指定位置。<br>   b. 编辑实体属性（位置、高度、朝向、工作参数等）。<br>   c. 支持批量部署和模板导入。<br>4. **路径规划**：<br>   a. 在地图上绘制飞行路径（点选航点、自由绘制）。<br>   b. 设置路径参数（高度、速度、动作等）。<br>   c. 支持多机路径协同编辑。<br>5. **作战行动定义**：<br>   a. 定义任务目标（目标实体、任务类型）。<br>   b. 定义作战行动序列（搜索、待机、攻击、评估等）。<br>   c. 设置行动触发条件（时间、事件）。<br>6. **场景验证**：<br>   a. 检查场景完整性（关键要素是否缺失）。<br>   b. 检查逻辑冲突（如路径穿越禁飞区、兵力超出可用数量）。<br>   c. 给出验证报告。<br>7. **保存与导出**：将场景保存至数据库，可导出为标准格式供其他模块使用。 |
| **成功判据**     | 1. 所有编辑操作（放置、移动、删除、属性修改）均能正确响应。<br>2. 地理信息加载正确，无偏移或错位。<br>3. 场景数据保存后与编辑时状态一致。<br>4. 验证功能能正确检测常见错误。<br>5. 导出的场景文件能被其他模块正确加载。 |
| **异常处理**     | 1. 地理信息加载失败：提示用户检查网络或数据源，可使用默认底图。<br>2. 模型库连接失败：提示用户使用本地缓存模型。<br>3. 拖拽放置时坐标超出合理范围：自动吸附到最近有效区域。<br>4. 路径绘制与地形冲突：提示用户调整高度。<br>5. 保存时数据冲突（多人同时编辑）：提示用户手动解决或自动合并。 |
| **后置条件**     | 1. 场景数据已保存至数据库，可供后续模块调用。<br>2. 编辑日志已记录。 |
| **显示约束**     | 1. 地图应支持缩放、平移、旋转等基本操作。<br>2. 实体应以图标或三维模型形式显示，不同类型有明显区分。<br>3. 路径应动态绘制，支持航点拖拽调整。<br>4. 支持图层管理（地形、影像、矢量、实体等显隐控制）。<br>5. 提供坐标显示和距离测量工具。<br>6. 支持多级撤销/重做。 |
| **其他约束**     | 1. 场景文件格式应标准化（如XML/JSON），便于交换。<br>2. 支持多用户协同编辑（可选）。<br>3. 编辑过程中应有自动保存机制（如每5分钟）。<br>4. 支持从外部导入标准格式（如KML、Shapefile）的地理要素。 |

## 4. 分析流程

### 4.1 图形化编辑核心机制
- 采用**模型-视图-控制器（MVC）**架构，地理信息作为底图视图，实体作为可拖拽的图形项。
- 实体放置：鼠标点击地图位置，从模型库选择实体类型，生成实体图形项，记录地理坐标。
- 路径绘制：支持航点点击生成、自由绘制、参数化生成（如圆形、螺旋线）。

### 4.2 地理信息集成
- 使用瓦片地图服务（TMS/WMTS）或本地瓦片缓存作为底图。
- 地形数据用于高程查询和地形跟随路径规划。
- 矢量图层用于显示边界、道路、河流等。

### 4.3 数据存储
- 场景数据以JSON/XML格式存储，包含：
  - 场景基本信息
  - 地理参考（坐标系、边界）
  - 实体列表（类型、ID、位置、属性）
  - 路径列表（航点、参数）
  - 作战行动定义（目标-行动关联）

# 反辐射无人机仿真系统——任务场景编辑模块开发实现方案

## 1. 总体架构

任务场景编辑模块采用**GIS引擎+图形化编辑器**架构，基于C++/Qt和地图渲染引擎（如Qt Location、MapLibre、Cesium for Qt），提供可视化的协同作战场景编辑环境。模块与模型库、数据管理模块紧密集成。

### 1.1 核心类设计
- **SceneManager**：场景管理器，维护所有场景元素（实体、路径、区域）。
- **MapViewController**：地图视图控制器，处理地图交互、图层管理。
- **EntityGraphicsItem**：实体图形项（QGraphicsItem派生），支持拖拽、选择、属性编辑。
- **PathGraphicsItem**：路径图形项，支持航点拖拽、曲线编辑。
- **GISEngine**：地理信息引擎，提供地图瓦片加载、坐标转换、高程查询。
- **SceneSerializer**：场景序列化/反序列化。

```cpp
class EntityGraphicsItem : public QGraphicsItem {
public:
    EntityGraphicsItem(const QString& modelPath, const QPointF& pos) {
        setFlag(QGraphicsItem::ItemIsMovable);
        setFlag(QGraphicsItem::ItemIsSelectable);
        setFlag(QGraphicsItem::ItemSendsGeometryChanges);
        setPos(pos);
        
        // 加载模型缩略图或简单图标
        m_pixmap = loadModelIcon(modelPath);
    }
    
    QRectF boundingRect() const override {
        return m_pixmap.rect();
    }
    
    void paint(QPainter* painter, const QStyleOptionGraphicsItem*, QWidget*) override {
        painter->drawPixmap(0, 0, m_pixmap);
        if (isSelected()) {
            painter->setPen(QPen(Qt::cyan, 2));
            painter->drawRect(boundingRect());
        }
    }
    
    QVariant itemChange(GraphicsItemChange change, const QVariant& value) override {
        if (change == ItemPositionChange) {
            QPointF newPos = value.toPointF();
            // 限制在地图范围内
            return scene()->sceneRect().contains(newPos) ? newPos : pos();
        }
        return QGraphicsItem::itemChange(change, value);
    }
    
private:
    QPixmap m_pixmap;
    QString m_entityId;
    QVariantMap m_properties;
};
```

### 3.3 路径规划与编辑

#### 3.3.1 路径图形项
```cpp
class PathGraphicsItem : public QGraphicsItem {
public:
    void addWaypoint(const QPointF& pos) {
        m_waypoints.append(pos);
        update();
    }
    
    QRectF boundingRect() const override {
        return shape().boundingRect();
    }
    
    QPainterPath shape() const override {
        QPainterPath path;
        if (m_waypoints.isEmpty()) return path;
        
        path.moveTo(m_waypoints.first());
        for (int i = 1; i < m_waypoints.size(); ++i) {
            path.lineTo(m_waypoints[i]);
        }
        return path;
    }
    
    void paint(QPainter* painter, const QStyleOptionGraphicsItem*, QWidget*) override {
        if (m_waypoints.size() < 2) return;
        
        painter->setPen(QPen(Qt::yellow, 2));
        for (int i = 0; i < m_waypoints.size() - 1; ++i) {
            painter->drawLine(m_waypoints[i], m_waypoints[i+1]);
        }
        
        // 绘制航点
        painter->setBrush(Qt::white);
        for (const auto& wp : m_waypoints) {
            painter->drawEllipse(wp, 4, 4);
        }
    }
    
private:
    QVector<QPointF> m_waypoints;
    double m_altitude;
    double m_speed;
};
```

#### 3.5.1 验证规则

```cpp
class SceneValidator {
public:
    ValidationResult validate(SceneManager* scene) {
        ValidationResult result;
        
        // 检查必需实体
        if (scene->getEntitiesByType("UAV").isEmpty()) {
            result.addError("未部署任何无人机");
        }
        if (scene->getEntitiesByType("Target").isEmpty()) {
            result.addWarning("未设置任务目标");
        }
        
        // 检查路径冲突
        auto conflicts = detectPathConflicts(scene);
        for (const auto& c : conflicts) {
            result.addError(QString("路径冲突: %1 与 %2").arg(c.id1).arg(c.id2));
        }
        
        // 检查威胁区域穿越
        auto threats = detectThreatCrossings(scene);
        for (const auto& t : threats) {
            result.addWarning(QString("路径穿越威胁区: %1").arg(t));
        }
        
        return result;
    }
};
```

由于采用了 OSGEarth，技术栈将从 Web 技术转向基于 C++ 的桌面应用开发，以获得更佳的渲染性能和更紧密的底层控制。

- **核心引擎**：**OSGEarth + OpenSceneGraph (OSG)**。OSGEarth 负责地理信息数据调度与渲染 ，OSG 负责底层三维场景管理和节点渲染 。
- **界面框架**：**Qt**。OSGEarth 与 Qt 的结合非常成熟，可以方便地将三维场景嵌入到 Qt 窗口中，并利用 Qt 的控件库构建强大的属性编辑面板和工具栏 。

- **数据存储**：采用 **XML** 作为场景描述文件（如 OSGEarth 原生支持的 `.earth` 文件），并辅以自定义的 **JSON** 或 **二进制文件** 存储动态的兵力部署和作战行动计划。矢量数据可直接使用 **Shapefile** 或 **GeoJSON** 。

### 二、基于 OSGEarth 的功能模块设计

#### 1. 核心概念映射：从实体到 OSGEarth 要素

OSGEarth 中的 **Feature**（要素）概念与我们之前定义的实体完美匹配。一个 `Feature` 由**矢量几何**（点、线、面）、**属性**（键值对）和**空间参考**组成 。我们可以通过 `Feature` 来实现所有作战实体。

| 仿真实体 | 几何表达 | OSGEarth 实现方式 | 关键属性（存储在Feature的属性中） |
| :--- | :--- | :--- | :--- |
| **任务目标** (雷达) | 点 | `osgEarth::Features::Feature` 创建点几何，并关联3D模型（如雷达模型）进行符号化替换 。 | 目标ID、类型、辐射频率、信号特征、优先级 |
| **威胁要素** (防空圈) | 面/体 | 创建多边形 `Feature`，通过 `altitude-clamping` 钳制到地形上，并设置半透明填充样式（`fill`）表示威胁范围 。 | 威胁类型、半径、杀伤概率、干扰强度 |
| **兵力部署** (无人机) | 点 | 与任务目标类似，使用点几何+3D模型（无人机模型）进行渲染。 | 无人机ID、型号、数量、燃油量、武器挂载 |
| **作战行动** (航线) | 线 | 创建线几何 `Feature`，设置样式（`stroke`、`stroke-width`），并通过 `altitude-clamping: terrain` 或 `absolute` 使其贴合地形或保持在绝对高度 。 | 航点列表（可存储在属性中）、速度、高度、动作 |

#### 2. 图形化场景编辑实现

- **场景构建与配置**：利用 OSGEarth 的 **`.earth` 文件**来配置基础地理环境 。用户通过界面操作，实际上是在后台动态生成或修改这个 XML 文件。例如，选择不同的影像底图（如卫星图）、高程数据（DEM），实际上是在 `.earth` 文件中添加或修改 `<image>` 和 `<elevation>` 图层 。

  ```xml
  <map name="Anti_Radiation_Scenario">
      <!-- 影像底图 -->
      <image name="world_image" driver="gdal">
          <url>../data/world.tif</url>
      </image>
      <!-- 高程数据 -->
      <elevation name="world_elev" driver="gdal">
          <url>../data/srtm.tif</url>
      </elevation>
      <!-- 后续动态添加的 Feature 层将会在此处引用 -->
  </map>
  ```

- **要素编辑交互**：
    - **绘制**：通过鼠标事件获取屏幕坐标，利用 `osgUtil::LineSegmentIntersector` 与 `MapNode` 进行求交，得到精确的地理坐标（经纬度、高程）。然后利用这些坐标生成 `Feature` 的点、线、面几何。
    - **属性编辑**：当用户选中一个 `Feature`（例如一个雷达模型）时，通过 Qt 信号触发属性面板更新。面板中的修改将直接写入 `Feature` 的属性字典（`Feature->attrs()`），并触发视图刷新。
    - **样式调整**：对于威胁范围等面要素，可以通过修改 `Style` 来实现 。例如，用户调整威胁半径时，程序会创建一个新的多边形几何，并重新应用包含 `fill` 和 `stroke` 定义的 `Style`。

#### 3. 作战行动规划：航线与协同

- **航路规划**：借鉴相关专利技术，利用 OSGEarth 的三维地形，可以开发自动或手动的航路规划功能 。
    - **手动规划**：用户在地形上点击生成一系列航点（`Waypoint`）。每个航点可以作为一个点 `Feature` 存储，包含经纬度、高度、速度等属性。连接这些航点的线即为航线。
    - **自动规划**：结合 `osgEarth::Util` 中的工具或集成第三方算法库，实现基于威胁规避、地形遮蔽的自动航线生成。例如，采用飞蛾扑火优化法等智能算法搜索三维最短或最安全的航路 。
- **协同关系可视化**：可以用不同颜色的连线（线 `Feature`）来表示无人机之间的协同关系（如编队、主从），并在线的属性中存储协同规则（如时间窗、相对位置）。

#### 4. 交互与界面

- **三维场景嵌入**：将 OSGEarth 的 `osgEarth::Qt::ViewerWidget` 嵌入到 Qt 的主窗口中，实现三维场景与 Qt 控件的无缝集成。
- **工具栏与面板**：利用 Qt 的 `QToolBar`、`QDockWidget` 等构建功能齐全的编辑器界面。
    - **实体创建工具栏**：点击“添加雷达”按钮，进入绘制模式。
    - **属性编辑器**：使用 `QTableWidget` 或 `QPropertyEditor` 展示和编辑选中 `Feature` 的属性。
    - **图层管理器**：以树形结构展示场景中的所有要素（目标、威胁、兵力），并控制其显隐、锁定、可编辑状态。这可以对应到 OSGEarth 中的不同 `FeatureModelLayer` 。

### 三、关键技术实现要点

1.  **要素的符号化与渲染**
    - **3D模型替换**：对于点状要素（无人机、雷达），使用 `ModelSymbol` 将点几何替换为外部加载的 3D 模型（如 `.osg`、`.ive`、`.gltf` 格式）。这可以通过在 `Style` 中设置 `model` 符号来实现 。
    - **样式动态更新**：威胁范围等要素需要根据属性动态改变样式。例如，威胁等级提高，其颜色从绿色变为红色。这可以通过在 `Style` 中使用**表达式**（`expression`）或 **JavaScript 回调**来实现，根据要素的属性值动态计算颜色值 。

2.  **三维交互与分析**
    - **地形跟随与钳制**：确保绘制的航线和部署的兵力自然地贴合或悬浮于地形之上。通过 `AltitudeSymbol` 的 `clamping` 属性（如 `terrain`、`relative`）和 `technique`（如 `map`、`gpu`、`drape`）来实现精确的地形交互 。
    - **可视域与通视分析**：利用 OSGEarth 的 `osgEarth::Util::LineOfSight` 工具，可以计算两点之间是否通视，这对于评估无人机侦察效果或通信链路至关重要。在编辑器中进行部署时，可以实时计算并显示通视情况 。

3.  **大规模场景管理**
    - 当作战场景包含大量要素（如数千个威胁点或复杂的航线网络）时，需要使用 OSGEarth 的**分页与动态调度**机制 。将要素数据组织成四叉树或 KD 树结构，根据视点的远近动态加载和卸载 `Feature`，确保渲染帧率稳定。

### 四、开发路径建议

1.  **环境搭建**：编译并集成 OSG、OSGEarth、Qt 及其依赖库（如 GDAL、GEOS），建立一个稳定的开发环境 。
2.  **原型验证**：创建一个基本的 Qt 应用程序，嵌入 OSGEarth 的 `ViewerWidget`，加载一个简单的 `.earth` 文件，验证三维地球显示正常。
3.  **核心编辑功能迭代**：
    - 实现鼠标点选地形并获取坐标。
    - 实现基于 `Feature` 和 `Style` 的简单点、线、面绘制和显示（例如，可以绘制一个黄色的威胁圆圈）。
    - 为绘制的要素添加属性编辑功能（如双击弹出属性框）。
    - 集成 3D 模型加载，替换简单的点要素。
4.  **专业功能深化**：开发航路规划、协同关系编辑、高级样式管理等复杂功能。
5.  **数据集成与交换**：定义清晰的场景数据存储格式（如自定义 XML），实现与仿真引擎的对接。
# 指标1 和 指标2区别
在反辐射无人机仿真系统中，这两个指标分别对应不同层次的功能模块，它们在职责和关注点上有着明确区分：

---

### 指标1：协同任务管理  
**功能定位**：**任务元数据管理与生命周期管理**  
- **管理对象**：任务的“档案”信息，即任务的**概要属性**。  
- **具体内容**：任务名称、编号、类型、目标（文字描述）、起止时间等**非空间、非图形化的基本信息**。  
- **操作范畴**：任务的创建、打开、保存、删除等文件级操作，相当于一个**任务清单或任务调度中心**。  
- **用途**：用于组织和管理多个任务，方便用户查找、加载、归档，属于系统级的基础管理功能。

---

### 指标2：任务场景编辑  
**功能定位**：**任务详细内容的图形化构建**  
- **管理对象**：任务的“蓝图”信息，即任务执行的**地理空间细节**。  
- **具体内容**：依托 GIS 地图编辑**任务背景**（如作战区域、地形）、**任务目标**（具体雷达/威胁的位置）、**威胁要素**（防空圈、干扰源）、**兵力部署**（无人机发射阵地、支援力量）、**作战行动**（航线、协同关系）等**图形化、空间化的内容**。  
- **操作范畴**：在地图上进行点、线、面绘制，属性编辑，模型放置，航路规划等**图形化编辑操作**。  
- **用途**：为仿真推演提供精确的战场环境和行动方案，属于核心的**专业编辑功能**。

---

### 二者关系  
- **层次不同**：任务管理是上层管理，场景编辑是下层细节填充。  
- **数据关联**：任务管理的元数据（如任务编号）通常作为场景文件的索引，一个任务对应一个场景文件。在任务管理中“打开”一个任务，系统会加载对应的场景文件进入编辑界面。  
- **分工明确**：任务管理确保用户可以高效组织多个作战想定，而场景编辑则提供了实现这些想定的可视化工具。

简单类比：任务管理如同“文件夹和文件名”，而场景编辑如同“文件内的具体文档内容”。两者共同构成完整的仿真任务准备系统。

# 主要实现
## 反辐射无人机协同作战想定详细展开

### 一、任务背景

**作战阶段**：战役发起前的“防空压制”阶段，旨在为后续有人机突击编队开辟安全走廊，削弱敌方一体化防空系统（IADS）的作战效能。

**战略目的**：通过反辐射无人机集群的持续压制与摧毁，使敌方关键预警雷达、火控雷达失能或被迫关机，从而夺取局部制空权，降低后续作战的突防风险。

**战场环境**：
- **地理**：作战区域为复杂丘陵地带，敌方防空阵地多部署于山谷或反斜面，利用地形掩护，传统反辐射导弹难以直接攻击。
- **气象**：中纬度地区，夏季，多云，能见度良好，但午后可能有局部对流天气影响无人机低空飞行。
- **电磁环境**：敌方拥有多种频段雷达（预警、目标指示、火控），并配备电子干扰系统，电磁频谱拥挤且动态变化。
- **敌我态势**：
    - **敌方**：以机动式防空导弹（如SA-10/20）为核心，辅以高炮和早期预警雷达，形成分层防御圈。雷达站点分布呈网络化，部分雷达具备间歇开关机能力。
    - **我方**：已通过侦察卫星、电子侦察机获取敌方雷达大致位置，但精确坐标及开机规律需临机确认。我方可投入反辐射无人机、电子战飞机、侦察无人机等多类平台。

### 二、任务目标

**目标清单**（按优先级排序）：
1. **早期预警雷达**（如P-18/19系列）：负责远程探测，是敌方防空系统的“眼睛”，优先摧毁可使敌方防御反应迟钝。
2. **多功能相控阵雷达**（如30N6E系列）：为SA-10/20提供目标照射和制导，是主要的威胁源，必须压制或摧毁。
3. **目标指示雷达**（如S波段搜索雷达）：为中程防空导弹提供目标指示，影响敌方火力通道数量。
4. **通信节点**（如战术数据链中继站）：切断敌防空系统的信息链路，使其陷入“盲聋”状态。

**目标特性**：
- **机动性**：部分雷达安装在机动底盘上，发射后可快速转移。
- **抗干扰能力**：具有频率捷变、低截获概率等能力。
- **防护手段**：可能配有近程防空武器（如“铠甲-S1”）保护雷达站点。

**预期效果**：
- 摧毁/压制80%以上的关键雷达节点，使敌方防空系统瘫痪或降级作战。
- 迫使剩余雷达频繁关机，丧失持续探测能力。
- 为后续有人机突防创造至少30分钟的无威胁窗口。

### 三、威胁要素

**敌方威胁**（按威胁等级排列）：

1. **防空火力威胁**：
   - **远程地空导弹**（SA-10/20等）：射程远（可达100km+），威胁高度覆盖广，是反辐射无人机的主要对手。无人机需利用低空突防和地形遮蔽规避。
   - **末端近防系统**（“铠甲-S1”、“道尔”等）：针对低空小型目标，射程虽近（10-20km），但反应快，对低飞无人机威胁极大。
   - **高炮**（ZSU-23-4等）：对于低速低空无人机仍有一定杀伤概率。

2. **电子干扰威胁**：
   - **有源干扰**：敌方可能释放大功率噪声干扰或欺骗干扰，阻塞无人机数据链或诱使其偏离目标。
   - **无源干扰**：投放箔条走廊，掩护雷达目标，干扰无人机对辐射源的定位。

3. **探测威胁**：
   - **低空补盲雷达**：专门探测低空小目标，反辐射无人机若飞行高度过低，可能被此类雷达提前发现。
   - **光电/红外探测**：部分近防系统配备光电跟踪设备，无源探测，反辐射无人机无法通过辐射引导规避。

4. **自然环境威胁**：
   - **地形**：丘陵地区可能产生多路径效应，影响无人机高度测量和导航精度。
   - **气象**：低云、降水可能影响无人机光电载荷效能，强风影响航线保持。

### 四、兵力部署

**我方兵力组成**：

| 兵力单元        | 型号/数量             | 部署位置             | 任务角色                       |
| ----------- | ----------------- | ---------------- | -------------------------- |
| **反辐射无人机群** | 哈比/ASN-301等，共计24架 | 前沿发射阵地（距前线80km）  | 主要打击力量，分波次出动，实施协同压制        |
| **侦察无人机**   | 1架（长航时，如“翔龙”）     | 战区高空巡航           | 实时侦察目标动态，确认雷达开机情况，引导反辐射无人机 |
| **电子战飞机**   | 2架（如“干扰-16”）      | 防区外盘旋            | 对敌方雷达实施远距离支援干扰，掩护无人机突防     |
| **地面指挥控制站** | 1座                | 后方隐蔽地域（距前线200km） | 接收情报，下达任务指令，监控作战进程         |
| **通信中继无人机** | 2架                | 中空盘旋，覆盖作战区域      | 保障无人机与指挥站之间的数据链畅通，克服地形遮挡   |

**部署原则**：
- 反辐射无人机发射阵地分散配置，避免被敌方一锅端。
- 电子战飞机与侦察无人机预先就位，形成“先侦后打、侦打一体”的态势。
- 通信中继无人机确保在复杂地形下各作战单元互联。

### 五、作战行动

**总体流程**：侦察定位 → 电子压制 → 无人机突防 → 协同攻击 → 战果评估 → 二次打击（如有必要）。

**详细行动方案**：

1. **预先准备阶段**
   - 侦察无人机升空，抵近侦察，确认目标雷达位置及工作频段。
   - 电子战飞机开始对敌方预警雷达实施间歇性干扰，消耗其抗干扰资源。
   - 地面指挥站根据最新情报，为每架反辐射无人机分配目标（目标分配基于优先级和无人机携带的导引头频段）。

2. **突防阶段**
   - 第一波12架反辐射无人机起飞，按照预先规划的**低空突防航线**飞行，利用地形遮蔽降低被探测概率。
   - 飞行中，无人机保持无线电静默，仅接收中继无人机转发的指令和更新目标数据。
   - 电子战飞机转为大功率压制，干扰敌方火控雷达和搜索雷达，制造“电磁迷雾”。
   - 若途中遭遇敌方防空导弹拦截，无人机可采取**机动规避**（如急转、俯冲）并释放诱饵。

3. **协同攻击阶段**
   - **时间协同**：各无人机根据到达目标区域的时间差，协调攻击时刻，实现**饱和攻击**，使敌方近防系统过载。
   - **空间协同**：多架无人机从不同方向同时进入目标区域，迫使敌方雷达分时应对，分散火力。例如，对某个关键雷达阵地，安排3架无人机分别从北、东、南三面同时抵近。
   - **任务协同**：部分无人机作为“诱饵”，故意暴露以吸引敌方近防系统开机，为后续真正的攻击无人机创造机会。诱饵机可配备角反射器增强雷达回波。
   - 当无人机进入目标辐射源探测范围后，其被动导引头开始截获雷达信号，若信号特征匹配，则锁定目标，转入**末制导**。
   - 攻击模式可选择**直接撞击**（针对高价值固定目标）或**徘徊待机**（针对可能关机的雷达，若信号消失则拉起巡航等待目标再次开机）。

4. **战果评估与二次打击**
   - 侦察无人机在攻击后抵近侦察，通过合成孔径雷达或光电设备评估毁伤效果。
   - 若目标未被完全摧毁，指挥站可指令第二波待命的12架无人机实施补充打击。
   - 同时，电子战飞机持续压制，防止敌方残存雷达组织反击。

5. **撤离阶段**
   - 攻击完毕且确认目标失能后，剩余无人机（若未自毁）可返航或按预设程序自毁于安全区域。
   - 各兵力单元按预案撤出作战区域，转入待战状态。

**应急措施**：
- 数据链中断：无人机按预编程自主飞行，直至恢复通信。
- 遭遇强干扰：无人机切换导引头频段或转入纯惯性导航，按预定坐标攻击。
- 目标关机：无人机在目标区域上空盘旋等待（如续航允许），若长时间不开机，则攻击预知的备用目标或返航。

---

上述展开详细描述了反辐射无人机协同作战的五个核心要素，体现了从战略意图到具体战术动作的完整链条，可为仿真系统的任务场景编辑提供详细的内容素材。


# 界面功能
## 反辐射无人机仿真系统——任务场景编辑器界面设计

### 一、设计目标
- 提供一个直观、高效、专业的图形化环境，用于构建反辐射无人机协同作战场景。
- 紧密结合 OSGEarth 三维地理信息系统，支持地形、影像加载与精确地理编辑。
- 采用 Qt 框架实现，遵循“所见即所得”原则，所有编辑操作实时反映在三维场景中。
- 界面布局清晰，功能分区合理，降低用户学习成本。

### 二、整体布局
采用经典的 **单文档界面（SDI）** 风格，主窗口包含以下核心区域：

```
+-----------------------------------------------------------------------------------+
| 标题栏                                                                             |
+-----------------------------------------------------------------------------------+
| 菜单栏：文件、编辑、视图、插入、工具、帮助                                          |

| |                   |                               |                            ||
```

### 三、各区域详细设计

#### ① 场景浏览器（左侧停靠面板）
以树形结构管理当前任务的所有要素，支持多标签页切换不同视图。

- **标签页1：任务属性**
  - 表单形式显示当前任务的元数据（对应“协同任务管理”功能）：
    - 任务名称（文本框）
    - 任务编号（文本框）
    - 任务类型（下拉框：防空压制/摧毁/侦察等）
    - 任务目标（多行文本，简要描述）
    - 起止时间（日期时间控件）
    - 备注（多行文本）
  - “应用”按钮保存修改。
  - 该面板内容与任务文件关联，切换任务时自动更新。

- **标签页2：图层管理器**
  - 按类别组织所有地理要素（目标、威胁、兵力、行动），每个类别前有复选框控制显隐。
  - 每个类别下可展开显示具体要素列表，右键菜单提供：
    - 缩放至要素
    - 删除
    - 复制
    - 属性编辑
  - 支持拖拽调整图层顺序（影响渲染顺序）。
  - 提供“添加图层”按钮，可导入外部矢量数据（Shapefile、KML）。

- **标签页3：要素列表**
  - 以表格形式列出所有要素（可搜索、排序）。
  - 双击某行快速定位并选中对应要素，同时右侧属性编辑器更新。

#### ② 属性编辑器（右侧停靠面板）
动态显示当前选中要素的详细属性，并根据要素类型自动切换表单内容。

- **通用属性**（所有要素均有）：
  - 名称、ID、描述、所属方（红/蓝/白）、可见性、锁定状态。

- **目标要素**（任务目标）：
  - 目标类型（雷达/防空导弹/指挥所）
  - 辐射特性：工作频段、功率、信号特征
  - 威胁等级（1~3级）
  - 优先级（1~5级）
  - 是否可移动（勾选框）
  - 关联的3D模型文件（文件选择器）

- **威胁要素**：
  - 威胁类型（防空火力/电子干扰/探测）
  - 威胁半径（数值+单位米）
  - 杀伤概率/干扰强度（滑块 0~1）
  - 颜色（颜色选择器，用于显示威胁范围）
  - 透明度（滑块）

- **兵力部署**：
  - 无人机型号（下拉框：预设型号列表）
  - 数量（整数微调框）
  - 燃油量/续航时间（数值+单位分钟）
  - 武器挂载（多选：反辐射导弹/诱饵/电子战吊舱）
  - 初始位置（经纬度、高程，可手动输入或从地图拾取）

- **作战行动**（航线/协同）：
  - 航线类型（攻击航线/巡逻航线/突防航线）
  - 航点列表（表格显示：序号、经度、纬度、高度、速度、动作）
    - 支持添加、删除、上移、下移航点
    - “从地图拾取”按钮添加当前鼠标位置为航点
  - 协同关系（表格显示：与本机协同的无人机ID、协同类型、触发条件、时间窗）
  - 飞行参数（巡航速度、最大速度、最低高度）

底部放置“应用”、“重置”按钮。

#### ③ 三维地图视图（中央区域）
嵌入 OSGEarth 的 `osgEarth::Qt::ViewerWidget`，作为主编辑区。

- **基础功能**：
  - 鼠标左键选择要素（拾取）。
  - 鼠标右键弹出上下文菜单（复制、删除、属性）。
  - 滚轮缩放，按住中键平移。
  - 支持多种导航模式（地球旋转、俯视、第一人称）。

- **绘制工具集成**：
  - 当工具栏选择绘制模式（点/线/面）时，鼠标在地图上点击生成对应几何体。
  - 绘制过程中实时显示坐标、长度、面积等信息。
  - 绘制完成后自动弹出属性编辑器。

- **三维特效**：
  - 威胁范围以半透明体（如圆锥、球体）显示，并随属性实时更新。
  - 无人机模型沿航线预览飞行（可播放动画）。
  - 地形分析（通视、剖面）结果可视化。

- **辅助显示**：
  - 左上角显示当前鼠标地理坐标、高程。
  - 右下角显示比例尺、指北针。
  - 支持切换底图（影像、矢量、地形）的浮动工具栏。

#### ④ 状态栏/底部面板
- **左侧**：实时提示当前操作状态（如“就绪”、“正在绘制航线...”）。
- **中间**：显示当前选择要素的信息摘要。
- **右侧**：进度条（用于加载大数据集时的反馈）。
- 可扩展的底部面板（默认隐藏）：点击“时序编辑器”按钮弹出，以甘特图形式展示各无人机任务起止时间，支持拖拽调整。

### 四、菜单与工具栏设计

#### 菜单栏
| 菜单 | 功能项 |
|------|--------|
| 文件 | 新建任务、打开任务、保存任务、另存为、导入(矢量/影像/高程)、导出(场景文件/报告)、退出 |
| 编辑 | 撤销、重做、剪切、复制、粘贴、删除、全选、查找要素 |

#### 工具栏
常用操作以图标按钮形式放置，分组排列：
- **文件操作**：新建、打开、保存。
- **编辑操作**：撤销、重做、删除。
- **选择操作**：选择（默认）、框选。
- **绘制操作**：点、线、面、矩形、圆形（用于威胁范围快速绘制）。

- **视图操作**：放大、缩小、漫游、全图。
- **播放控制**（用于预览）：播放航线、暂停、停止。

### 五、交互流程示例（添加一个威胁要素）
1. 用户点击工具栏上的“绘制圆形”按钮（代表威胁范围）。
2. 鼠标移动到三维地图上，左键单击确定圆心，拖动确定半径，再次单击完成绘制。
3. 绘制完成后，自动弹出右侧属性编辑器，并默认选中刚绘制的威胁要素。
4. 用户在属性编辑器中设置威胁类型、半径精确值、颜色、透明度等。
5. 点击“应用”，场景中的圆形区域立即更新样式。
6. 同时，左侧场景浏览器的图层管理器“威胁”节点下新增一个要素，名称可编辑。

### 六、数据交换与扩展性
- 所有编辑内容实时保存在内存中，只有用户点击“保存”时才写入文件。
- 文件格式采用自定义 JSON 或 XML，包含任务元数据、要素列表（几何+属性）、样式定义。
- 支持将场景导出为仿真引擎可读的格式（如特定 XML 或二进制）。
- 预留插件接口，方便日后增加新的要素类型或分析工具。

### 七、界面风格建议
- 采用深色主题（军事软件常用），降低视觉疲劳，突出三维场景。
- 图标使用清晰、直观的符号（如雷达图标代表目标，盾牌代表威胁，无人机图标代表兵力）。
- 字体统一，确保在高分辨率显示器上清晰。

此设计方案充分结合了 OSGEarth 的三维能力与 Qt 的界面灵活性，为反辐射无人机仿真任务场景编辑提供了完整、专业的解决方案。
