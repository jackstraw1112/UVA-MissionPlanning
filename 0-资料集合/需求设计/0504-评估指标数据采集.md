# 反辐射无人机仿真系统——作战效能评估模块评估指标数据采集功能需求描述

## 1. 功能需求描述

| **项目**         | **描述** |
|------------------|----------|
| **需求名称**     | 评估指标数据采集 |
| **需求标识**     | FR-EVAL-04 |
| **需求概述**     | 支持仿真过程中自动采集与评估指标相关的各类数据，对采集到的原始数据进行转换、汇总、标准化等预处理操作，为后续效能评估计算提供规范化的数据输入。 |
| **需求来源**     | 系统总体设计要求 / 作战效能评估模块能力指标 |
| **前置条件**     | 1. 仿真任务已启动，并处于运行状态。<br>2. 评估指标体系已构建，明确所需采集的指标及其计算所需数据项。<br>3. 数据采集模块已配置好采集参数（采集频率、数据源等）。 |
| **触发条件**     | 1. 仿真开始运行，自动触发数据采集。<br>2. 用户手动启动数据采集（可选）。<br>3. 达到预设的采集时间点或事件触发。 |
| **退出条件**     | 1. 仿真结束，数据采集自动停止。<br>2. 用户手动停止采集。<br>3. 系统异常导致采集中断。 |
| **功能参与者**   | 数据采集服务（后台）、仿真引擎（数据源）、预处理模块、评估计算模块、用户（监控） |
| **输入**         | 仿真过程中产生的原始数据（如实体状态、事件日志、武器发射记录、毁伤结果等），数据采集配置参数（采集频率、指标映射规则）。 |
| **输出**         | 经过预处理后的规范化指标数据集（可用于评估计算），包括时间序列数据、统计汇总数据等。 |
| **处理流程**     | 1. **配置解析**：启动时读取指标映射规则，确定需要采集的数据字段、计算周期、预处理方式。<br>2. **数据捕获**：在仿真运行过程中，按设定频率（如每秒一次）从仿真引擎的数据接口捕获原始数据，或订阅仿真事件流实时接收。<br>3. **数据转换**：将原始数据格式转换为内部统一格式，根据指标定义提取或计算所需字段。<br>4. **数据汇总**：对连续时间序列数据进行统计汇总（如求和、平均值、最大值、最小值等），可按任务阶段或时间窗口聚合。<br>5. **数据标准化**：对汇总后的数据进行标准化处理（如归一化、标准化得分），消除量纲影响。<br>6. **存储与推送**：将预处理后的数据存入实时数据库或内存缓冲区，同时可推送给评估计算模块进行在线评估。 |
| **成功判据**     | 1. 能正确采集仿真过程中所有预定义的指标相关数据。<br>2. 数据转换准确，无信息丢失。<br>3. 汇总和标准化结果符合预设规则，数学上正确。<br>4. 数据采集与仿真运行同步，无显著延迟。 |
| **异常处理**     | 1. 数据源不可用或连接中断：记录错误，尝试重连，超时后停止采集并告警。<br>2. 采集频率过高导致性能下降：动态调整采样频率或给出性能警告。<br>3. 数据格式不匹配：跳过该条数据，记录日志，继续后续采集。<br>4. 标准化计算异常（如除零）：使用默认值或标记为无效数据。 |
| **后置条件**     | 1. 预处理后的指标数据集可供评估模块随时查询。<br>2. 采集过程日志记录完备，便于事后审计。 |
| **显示约束**     | 1. 用户可实时查看当前采集状态（采集频率、已采集数据量）。<br>2. 可配置采集参数界面，支持采集规则的增删改查。<br>3. 可预览最近采集的原始数据和预处理结果。 |
| **其他约束**     | 1. 数据采集不应显著影响仿真运行性能，建议采用异步非阻塞方式。<br>2. 支持多种数据源接口（共享内存、网络、数据库）。<br>3. 预处理规则应可扩展，支持用户自定义转换函数。 |

## 2. 关键术语解释

| **术语** | **解释** |
|----------|----------|
| **数据采集** | 从仿真引擎或其他数据源获取原始数据的过程，通常按照一定频率或事件触发进行。 |
| **转换** | 将原始数据格式转换为评估模块可识别的内部格式，可能包括单位换算、字段提取、类型转换等。 |
| **汇总** | 对一组数据（如一个时间窗口内的多个采样点）进行统计计算，得到总和、平均值、最大值、最小值等聚合结果。 |
| **标准化** | 将数据按一定规则缩放，消除量纲影响，常见方法包括min-max归一化、z-score标准化等，使不同指标的数据具有可比性。 |

## 4. 分析流程

### 4.1 数据采集流程
1. **初始化阶段**：解析指标映射规则，确定需要采集的数据项（如每架无人机的坐标、速度、武器发射时刻、命中事件等）。建立与仿真引擎的数据通道。
2. **采集循环**：
   - 定时器触发或事件驱动，从数据通道获取最新数据。
   - 对每条数据，根据映射规则提取相关字段，进行必要的格式转换（如将经纬度转换为平面坐标）。
   - 将转换后的数据暂存于环形缓冲区。
3. **汇总处理**：
   - 根据预设的时间窗口（如每10秒、每分钟）或任务阶段（如搜索阶段、攻击阶段），对缓冲区内的数据进行聚合计算。
   - 例如，计算某架无人机在攻击阶段的平均速度、最大加速度等。
4. **标准化处理**：
   - 对汇总后的指标值进行标准化。例如，使用min-max归一化：`x_norm = (x - min) / (max - min)`，其中min和max可全局指定或从历史数据中动态获取。
   - 若指标方向为负向（越小越好），可进行逆向处理（如取倒数或`1-x`）。
5. **存储/输出**：将标准化后的指标数据存入时间序列数据库，同时可实时推送给评估模块。

### 4.2 性能考虑
- 采用生产者-消费者模型，采集线程与预处理线程分离，避免阻塞。
- 使用内存映射文件或共享内存实现高效数据传输。
- 对实时性要求不高的汇总任务，可定时批量处理。

# 反辐射无人机仿真系统——评估指标数据采集模块开发实现方案

## 1. 总体架构

评估指标数据采集模块作为仿真引擎与效能评估模块之间的桥梁，采用**生产者-消费者**异步架构，确保高性能、低延迟的数据处理。模块与仿真引擎解耦，支持多种数据源接口，并提供可扩展的数据预处理管道。

```plaintext
┌─────────────────┐      ┌─────────────────────────────────────────────┐      ┌─────────────────┐
│  仿真引擎        │      │             数据采集模块                      │      │  评估计算模块   │
│  (数据源)        │ ──→ │  ┌─────────┐ ┌─────────┐ ┌─────────┐       │ ──→ │  (指标使用方)   │
└─────────────────┘      │  │采集适配器│→│转换管道 │→│汇总标准化│       │      └─────────────────┘
                          │  └─────────┘ └─────────┘ └─────────┘       │
                          │           环形缓冲区/数据库                   │
                          └─────────────────────────────────────────────┘
```

### 1.1 模块层次结构
- **采集适配器层**：对接不同数据源（共享内存、ZeroMQ、文件），实现数据接收。
- **管道处理层**：包含转换器、汇总器、标准化器，每个阶段可配置并可并行。
- **存储层**：环形缓冲区（实时）、时间序列数据库（持久化）。
- **管理监控层**：配置管理、性能监控、日志记录。

## 2. 数据采集实现

### 2.1 数据源接口
支持多种数据源接入，通过抽象工厂模式创建适配器：
```cpp
class DataSourceAdapter {
public:
    virtual bool connect() = 0;
    virtual bool startAcquisition(std::function<void(const RawData&)> callback) = 0;
    virtual void stop() = 0;
};

// 共享内存适配器
class SharedMemoryAdapter : public DataSourceAdapter { ... };
// ZeroMQ适配器（用于分布式仿真）
class ZMQAdapter : public DataSourceAdapter { ... };
// 文件回放适配器（用于事后分析）
class FileReplayAdapter : public DataSourceAdapter { ... };
```

### 2.2 数据接收
- 采用**异步非阻塞I/O**，避免阻塞仿真引擎。
- 使用**无锁队列**（如 `moodycamel::ConcurrentQueue`）暂存原始数据，生产者线程（数据源回调）入队，消费者线程（预处理）出队。
- 支持**高频率采集**（≥100Hz），通过调整消费者线程数保证处理能力。

### 2.3 数据格式
原始数据统一封装为 `RawData` 结构，包含时间戳、实体ID、数据类型（位置、事件等）、JSON格式的负载。
```cpp
struct RawData {
    uint64_t timestamp;      // 微秒级时间戳
    uint32_t entityId;
    DataType type;           // 枚举：POSITION, EVENT, WEAPON, etc.
    QJsonObject payload;      // 具体数据
};
```

## 3. 数据转换实现

### 3.1 映射规则配置
使用JSON文件定义指标映射规则，例如：
```json
{
  "indicators": [
    {
      "name": "kill_prob",
      "source": "event",
      "condition": "type == 'weapon_hit'",
      "fields": {
        "hit_count": "count()",
        "total_shots": "sum(launch_count)"
      },
      "transform": "hit_count / total_shots"
    }
  ]
}
```

### 3.2 转换引擎
- 采用**轻量级脚本引擎**（如 Lua 或嵌入式 JavaScript）执行转换逻辑，支持自定义函数。
- 或使用**表达式解析库**（如 `muParser`）进行简单计算。
- 转换器从队列中取出原始数据，根据规则提取字段，生成 `ProcessedDataItem`。

### 3.3 批量处理优化
- 每次从队列中取出批量数据（例如1000条）进行转换，减少函数调用开销。
- 使用 SIMD 指令加速数值转换（如经纬度转平面坐标）。

## 4. 数据汇总实现

### 4.1 时间窗口策略
- **固定窗口**：按固定时间长度（如10秒）划分，每个窗口独立计算。
- **滑动窗口**：每步进（如1秒）计算过去N秒的数据，用于实时监控。
- **事件窗口**：按任务阶段（如“攻击阶段”）由仿真事件触发开始/结束。

### 4.2 汇总算法
- 使用**滑动窗口累加器**（如 `boost::accumulators`）高效维护均值、方差、极值等统计量。
- 示例：计算某指标在窗口内的平均值
```cpp
class SlidingWindowAccumulator {
    std::deque<double> buffer;
    double sum = 0;
    size_t capacity;
public:
    void add(double value) {
        buffer.push_back(value);
        sum += value;
        if (buffer.size() > capacity) {
            sum -= buffer.front();
            buffer.pop_front();
        }
    }
    double mean() const { return sum / buffer.size(); }
};
```

### 4.3 并行汇总
- 不同指标可并行汇总，使用 `QtConcurrent::map` 或 OpenMP。
- 对于同一指标的不同实体（如240架无人机），可按实体分片处理。

## 5. 数据标准化实现

### 5.1 标准化方法
- **Min-Max归一化**：`x_norm = (x - min) / (max - min)`
- **Z-Score标准化**：`x_norm = (x - mean) / stddev`
- **百分比排名**：用于非参数化比较。

### 5.2 参数动态更新
- min/max/mean/stddev 可从历史数据中动态计算，或由用户预先设定。
- 使用**在线算法**（Welford算法）高效计算均值和方差，避免存储全部历史。

### 5.3 实现示例
```cpp
class Normalizer {
    double min_, max_;
public:
    double normalize(double x) const {
        if (max_ - min_ < 1e-9) return 0.5; // 防除零
        return (x - min_) / (max_ - min_);
    }
    void updateMinMax(double x) {
        min_ = std::min(min_, x);
        max_ = std::max(max_, x);
    }
};
```

## 6. 性能优化策略

| **优化项** | **实现方法** | **预期效果** |
|------------|--------------|--------------|
| 无锁队列 | 使用 `moodycamel::ConcurrentQueue` | 避免锁竞争，提升吞吐量 |
| 批量处理 | 每次从队列取一批数据（如1000条） | 减少函数调用次数，提升缓存局部性 |
| 内存池 | 预分配 `ProcessedDataItem` 对象池 | 减少动态内存分配 |
| 流水线并行 | 转换、汇总、标准化三个线程流水线 | 提高CPU利用率 |
| SIMD优化 | 对标准化等批量数值操作使用SIMD指令 | 加速3~5倍 |
| 数据压缩 | 存储时使用Snappy或LZ4压缩 | 降低I/O压力 |

## 7. 与评估模块的接口

### 7.1 实时推送
- 标准化后的数据通过 `Qt信号槽` 或 `ZeroMQ` 实时推送给评估计算模块。
- 支持订阅模式：评估模块可订阅特定指标的数据流。

### 7.2 查询接口
- 提供RESTful API或共享内存查询接口，允许评估模块按时间范围拉取历史数据。
- 数据存储在 **InfluxDB** 或 **TimescaleDB** 中，便于时间序列查询。
