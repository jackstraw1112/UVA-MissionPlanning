## 1. 功能需求描述

| **项目**         | **描述** |
|------------------|----------|
| **需求名称**     | 试验数据管理 |
| **需求标识**     | FR-DATA-02 |
| **需求概述**     | 具备试验数据增加、删除、修改、查询等管理功能，支持用户对试验数据（包括原始采集数据、清洗后数据、预处理中间结果等）进行全生命周期的维护，确保数据可追溯、可操作。 |
| **需求来源**     | 系统总体设计要求 / 试验数据处理模块能力指标 |
| **前置条件**     | 1. 用户已通过身份认证并具有试验数据管理权限。<br>2. 试验数据已成功导入数据库或文件系统。<br>3. 系统数据库连接正常。 |
| **触发条件**     | 1. 用户进入“试验数据管理”界面，执行相应操作（增、删、改、查）。<br>2. 用户通过工具栏或菜单选择相应功能。 |
| **退出条件**     | 1. 用户关闭试验数据管理界面。<br>2. 操作完成，系统给出反馈。 |
| **功能参与者**   | 用户（试验分析人员）、数据库系统、文件系统（若存储外部文件） |
| **输入**         | 查询条件（如时间范围、数据类型、关键词）、新增/修改的数据记录、删除确认指令。 |
| **输出**         | 数据列表（表格）、单个记录的详细信息、操作结果提示（成功/失败）。 |
| **处理流程**     | 1. **查询**：用户输入查询条件，点击“查询”，系统根据条件从数据库检索数据，并以表格形式展示，支持分页、排序。<br>2. **增加**：用户点击“新增”，在弹出的对话框中填写数据字段，提交后系统进行格式校验，若通过则插入数据库，刷新列表并显示成功提示。<br>3. **修改**：用户选中某条记录，点击“修改”，弹出预填数据的编辑框，修改后提交，系统校验并更新数据库，刷新列表。<br>4. **删除**：用户选中一条或多条记录，点击“删除”，系统弹出确认对话框，确认后执行删除操作，从数据库移除记录（可选逻辑删除），刷新列表并显示成功提示。<br>5. **批量操作**：支持批量导入、导出、删除。 |
| **成功判据**     | 1. 查询结果准确，能正确返回符合条件的数据。<br>2. 新增数据能正确写入数据库，并在列表中可见。<br>3. 修改数据能正确更新数据库，原数据被覆盖。<br>4. 删除数据后，数据库中不再存在该记录（或标记为删除且不再显示）。<br>5. 所有操作均有明确的成功/失败反馈。 |
| **异常处理**     | 1. 输入数据格式不正确（如日期格式错误、必填字段为空）：弹出提示，高亮错误字段，拒绝提交。<br>2. 数据库连接失败：提示“数据库连接异常，请稍后重试”，并记录错误日志。<br>3. 删除数据时若有关联数据（如外键约束），提示“该数据已被其他记录引用，无法删除”，并列出引用信息。<br>4. 批量操作时部分成功部分失败：记录失败详情，提示用户查看日志。 |
| **后置条件**     | 1. 数据库内容已根据操作同步更新。<br>2. 操作记录写入系统日志，便于审计。 |
| **显示约束**     | 1. 数据列表应支持分页、排序、过滤功能。<br>2. 列表每行应提供编辑、删除按钮（或通过右键菜单）。<br>3. 新增/编辑对话框应包含所有字段，并标注必填项。<br>4. 支持选中多行进行批量删除。 |
| **其他约束**     | 1. 试验数据应包含版本号或时间戳，便于追溯。<br>2. 删除操作建议采用逻辑删除（标记删除）而非物理删除，除非用户明确要求物理删除。<br>3. 支持按用户权限控制数据可见性（如某些敏感数据需特殊权限）。 |

## 4. 分析流程

试验数据管理本质是对数据库表的常规CRUD操作，但在仿真系统中需考虑数据量大、版本控制、数据关联等因素。典型流程如下：

- **查询**：根据用户输入构造SQL查询（如 `SELECT * FROM test_data WHERE ...`），执行并返回结果集。可结合分页（`LIMIT offset, count`）和排序（`ORDER BY`）。
- **增加**：对于单条增加，使用 `INSERT INTO ... VALUES ...`。对于批量增加（如文件导入），可解析文件后构建批量插入语句，使用事务提高效率。
- **修改**：先查询原记录（可选），然后构造 `UPDATE ... SET ... WHERE id = ...` 语句执行。
- **删除**：若为物理删除，执行 `DELETE FROM ... WHERE id IN (...)`。若为逻辑删除，执行 `UPDATE ... SET deleted=1 WHERE id IN (...)`，并在查询时增加 `WHERE deleted=0` 条件。

为防止并发冲突，可在数据表中增加版本号字段，更新时检查版本号。

# 反辐射无人机仿真系统——试验数据管理模块开发实现方案

## 1. 总体架构

试验数据管理模块采用**MVC模式**，后端基于C++/Qt和SQL数据库（如SQLite/PostgreSQL），前端使用Qt Widgets实现数据表格、编辑对话框等交互界面。核心组件包括：

- **数据模型层**：封装数据库操作，提供CRUD接口。
- **视图层**：`QTableView` 或 `QTableWidget` 显示数据，支持排序、过滤、分页。
- **控制器层**：处理用户操作，调用模型方法，更新视图。
- **数据库连接池**：管理多线程并发访问。

## 2. 核心技术选型

| **组件**       | **技术方案**                          | **说明**                               |
|----------------|---------------------------------------|----------------------------------------|
| 数据库         | SQLite (单机) / PostgreSQL (网络)      | 根据部署规模选择                       |
| ORM            | Qt SQL 模块 (`QSqlDatabase`, `QSqlQuery`) | 轻量级，直接使用SQL                    |
| 数据表格       | `QSqlTableModel` + `QTableView`        | 实现自动的数据绑定和编辑功能            |
| 分页           | 自定义 `LIMIT` + `OFFSET`               | 或使用游标分页                         |
| 排序           | `ORDER BY` + 点击表头                   | 通过 `QHeaderView` 信号触发             |
| 过滤           | `WHERE` 子句 + 用户输入                  | 可结合 `QSortFilterProxyModel`          |
| 批量操作       | 事务 + 批量SQL                           | 提升效率                               |

## 3. 功能实现细节

### 3.1 查询功能

- **界面**：提供查询条件输入区域（如时间范围、数据类型选择、关键词输入框），点击“查询”按钮触发。
- **后端**：根据条件动态构造SQL的`WHERE`子句，使用 `QSqlQuery` 执行，结果通过 `QSqlQueryModel` 或自定义模型展示在 `QTableView` 中。
- **分页**：记录总数，每次查询只加载当前页数据，使用 `LIMIT pageSize OFFSET (page-1)*pageSize`。
- **排序**：监听表头点击信号，获取列名和排序方向，重新组织SQL的 `ORDER BY` 并刷新数据。
- **索引优化**：对经常查询的字段（如时间、数据类型）建立数据库索引，加速查询。

### 3.2 增加功能

- **单条增加**：弹出编辑对话框，用户输入字段后，点击“保存”触发插入。后端使用 `INSERT INTO ... VALUES (...)`，成功后刷新列表。
- **批量增加**：支持从文件导入（CSV/Excel），解析后使用事务一次性插入多条记录，减少事务开销。代码示例：
  ```cpp
  QSqlDatabase::database().transaction();
  for (const auto& record : records) {
      QSqlQuery query;
      query.prepare("INSERT INTO test_data (field1, field2) VALUES (?, ?)");
      query.addBindValue(record.field1);
      query.addBindValue(record.field2);
      query.exec();
  }
  QSqlDatabase::database().commit();
  ```

### 3.3 修改功能

- **编辑**：选中一行，点击“编辑”或双击行，弹出编辑对话框，预填当前数据。用户修改后提交，后端使用 `UPDATE ... SET ... WHERE id = :id` 更新。
- **并发控制**：若多人同时修改同一条数据，可采用乐观锁（增加版本号字段）。更新时检查版本号，若版本号已变，提示用户冲突。

### 3.4 删除功能

- **物理删除**：执行 `DELETE FROM test_data WHERE id IN (...)`，直接从数据库移除。适用于临时数据。
- **逻辑删除**：增加 `deleted` 布尔字段，删除时执行 `UPDATE test_data SET deleted=1 WHERE id IN (...)`，查询时默认加上 `WHERE deleted=0`。可提供“回收站”功能恢复数据。

## 4. 性能优化措施

| **优化项**         | **实现方法**                                                 | **预期效果**                     |
|--------------------|--------------------------------------------------------------|----------------------------------|
| 数据库索引         | 对查询条件字段（时间、数据类型）建立索引                     | 查询速度提升10倍以上             |
| 分页加载           | 避免一次性加载所有数据，每次只加载当前页                     | 内存占用降低，首屏加载快          |
| 连接池             | 使用 `QSqlDatabase` 的克隆连接，避免频繁打开关闭连接         | 减少连接开销                     |
| 批量操作           | 插入/删除/更新使用事务，合并SQL语句                           | 批量操作速度提升5-10倍            |
| 缓存               | 对常用数据（如数据类型字典）使用内存缓存                      | 减少数据库查询                    |
| 异步查询           | 使用 `QtConcurrent` 在后台线程执行耗时查询，避免界面卡死      | 界面保持流畅                     |

## 6. 关键技术难点与解决方案

| **难点**                     | **解决方案**                                             |
|------------------------------|----------------------------------------------------------|
| 大数据量下查询缓慢           | 建立索引、分页、使用 `EXPLAIN` 分析查询计划优化           |
| 并发写入冲突                 | 乐观锁（版本号）或悲观锁（数据库行锁）                    |
| 批量导入性能                 | 使用事务、预处理语句、调整提交频率                        |
| 数据软删除与关联数据一致性   | 外键设为 `SET NULL` 或 `CASCADE`，或应用层处理           |
| 用户权限控制                 | 在数据模型中添加权限过滤，根据用户角色隐藏部分数据        |
